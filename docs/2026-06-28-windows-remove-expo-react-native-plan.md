# Windows Expo and React Native Removal

## Summary

The Windows app has moved from Expo Router/Metro and the React Native
compatibility layer to Vite, TanStack Router, Pickle UI, and Tailwind. The
companion app stays Expo/React Native and continues to use `@pane/ui`.

## Shipped Outcome

- `apps/windows` depends on React, React DOM, Vite, TanStack Router, TanStack
  Query, Tauri APIs, Pickle UI, Tailwind, and web-only utilities.
- `apps/windows` has no dependency on Expo, React Native, React Native Web,
  Tamagui, native slider packages, React Native SVG, or `@pane/ui`.
- Windows imports Pickle primitives directly from `pickle-ui`.
- Windows Tailwind/Pickle CSS flows through
  `apps/windows/src/styles/windows.source.css` and the first-party
  `@tailwindcss/vite` plugin. There is no checked-in or locally generated
  `windows.generated.css` artifact.
- TanStack Router `autoCodeSplitting` is enabled so route components build into
  lazy chunks without hand-written `.lazy.tsx` route files.
- Public Windows routes remain unchanged.

## Current Windows Shape

- Entrypoint: `apps/windows/src/main.tsx`.
- Routes: `apps/windows/src/routes/`.
- Route tree: `apps/windows/src/routeTree.gen.ts`, generated by the TanStack
  Router Vite plugin and kept checked in so `tsc --noEmit` works from a fresh
  checkout.
- Global CSS: `apps/windows/src/styles/global.css`,
  `apps/windows/src/styles/shell.css`, and
  `apps/windows/src/styles/windows.source.css`.
- Fonts load through `apps/windows/src/lib/register-pane-fonts.ts` using
  `@fontsource/geist` and `@fontsource/jetbrains-mono`.

## Completed Work

- Removed `UIProvider` and `@tamagui/web/reset.css` from the Windows root.
- Replaced Windows `@pane/ui` usage with Pickle UI and web-native React code.
- Removed Windows native-module shims for native slider and React Native SVG.
- Removed React Native aliases from `apps/windows/vite.config.ts`.
- Removed stale `apps/windows/app/` leftovers.
- Removed Windows direct dependencies on `@pane/ui`, Expo, Metro, React Native,
  React Native Web, Tamagui, native slider, React Native SVG, Tabler icons, and
  `qrcode.react`.
- Replaced the Tailwind CLI prebuild step with `@tailwindcss/vite`, so Vite
  handles Tailwind compilation directly during dev and build.
- Enabled TanStack Router route-component code splitting, reducing the main
  production JS chunk from roughly 532 kB to roughly 353 kB in the local build.

## Companion Boundary

Do not migrate the companion app as part of this work.

- `apps/mobile` remains Expo/React Native.
- `packages/ui` remains available for `apps/mobile`.
- React Native, Tamagui, or Expo dependencies that are still used by
  `apps/mobile` or `packages/ui` should remain in the root lockfile.

## Acceptance Criteria

- `rg -n "from ['\"]@pane/ui['\"]|@pane/ui" apps/windows` returns no active
  Windows app imports.
- `rg -n "react-native|react-native-web|@tamagui|tamagui|expo|metro"`
  `apps/windows apps/windows/package.json apps/windows/vite.config.ts` returns
  no active Windows runtime/build references, except historical docs or
  companion-specific text outside `apps/windows`.
- `apps/windows/package.json` has no `@pane/ui`, Expo, Metro, React Native,
  React Native Web, Tamagui, native slider, or React Native SVG dependency.
- Vite build no longer needs Windows native-module shims or a Tailwind CLI
  prebuild artifact.
- Main-window route components are code-split by the router plugin.
- The companion app still typechecks.

## Verification

```powershell
npm install

npm run format:check
npm run build -w @pane/windows
npm run rust:fmt:check
npm run tauri:gen:check
```

Manual smoke test with `npm run dev`:

- Main window boots and reveals correctly.
- Sidebar navigation works for all dashboard modules.
- Capture fullscreen, area selector, floating preview, zoom window, and image
  editor still work.
- Accent popup opens and receives initial `chars` query data.
- Companion page displays the pairing QR code and trusted-device states.
- Dev updater state still skips update checks.
- Child windows still load direct route paths through
  `child_webview_url::routes`.
